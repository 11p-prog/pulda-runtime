# ADR-0001 — Adopt Workspace model over Dashboard

## Status
Accepted

## Context
The first UI pass (Stitch design) followed a conventional dashboard/todo
pattern: greeting hero, widget tiles, checklist. In review, this was
identified as fundamentally mismatched with what Pulda is trying to be —
a system for operating across life domains and time, not a task tracker.
The concrete symptom was: users couldn't tell *where* they were working
(which domain, which point in time) without extra explanation, because a
dashboard has no persistent notion of "current workspace."

## Decision
Pulda's UI is structured as a Workspace, not a Dashboard: the center pane
always represents one specific (Domain Context × Time Context)
combination, switching between them is an explicit navigation action (tabs
+ calendar), and nothing is shown as a floating "insight" disconnected
from a specific context. Formalized as: "Pulda does not visualize
widgets. Pulda visualizes the user's current operating context."

## Consequences
- Every new screen/section must be justified by which workspace context it
  belongs to — a widget that doesn't represent the user's actual current
  state (e.g. a zero-data Health metric) is now recognized as a design
  defect (see CR-0001, the Health-widget removal under CR-0002/#3), not a
  style choice.
- Date and Domain became two independent, explicit context axes (see
  ADR-0003 and CR-0002) instead of a single "current view" concept.
- Historical Import (CR-0005) is defined as producing Events inside this
  same workspace model, not a separate "imported documents" view.
