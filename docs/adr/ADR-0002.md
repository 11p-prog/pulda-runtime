# ADR-0002 — Event First as the core data model

## Status
Accepted

## Context
Early UI treated captured items as todo-list line items — status
checkboxes, a single flat list. But Pulda's product goal is broader:
capture now, classify/interpret later, and let items evolve through a
lifecycle (captured → classified → in progress → done/deferred/blocked)
across many life domains, not just "tasks to finish."

## Decision
Every piece of input to Pulda is an Event first — a single, uniform
record with free-text content, a capture timestamp, and a role/area
classification — before it is ever treated as a task, a journal entry, a
decision, or anything else. The UI communicates this lifecycle
explicitly (status labels, Waiting/Overdue/Deferred as *states* of an
Event, not separate item types) rather than presenting Pulda as a
checklist app.

## Consequences
- The capture bar is the primary interaction surface, always available,
  not a buried "add task" form (CR-0001).
- Historical import (CR-0005) must produce ordinary Events through the
  same `create_event` path — no parallel "imported document" data model.
- Classification (rule-based today, per CR-0006/RFC-0001) is a step that
  happens to an Event after capture, never a precondition for capturing.
