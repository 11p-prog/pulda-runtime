# ADR-0003 — Calendar is Navigation, not the primary workspace

## Status
Accepted

## Context
Before this decision, clicking a calendar date only highlighted a cell —
the center content stayed on "today" regardless of the date clicked. This
made the calendar feel decorative, and left no way to actually revisit a
past day's real record.

## Decision
The calendar's role is strictly navigation: it lets the user find a date.
Selecting a date is an explicit workspace switch — "Navigation exists to
find. Workspace exists to work." — that activates that date's real
workspace (that day's plan/stats/events/review), scoped by the shared
`context_workspace(ctx, selected_date)` function (CR-0002). The calendar
itself never becomes the primary work surface; it only routes into one.

## Consequences
- Date became a second independent context axis alongside Domain (see
  ADR-0001), not a filter parameter bolted onto a single dashboard view.
- A selected historical date shows what actually happened that day (a
  record), and intentionally hides live-only state (Waiting/Overdue/
  Deferred) that only makes sense for the live Today view (CR-0002).
- "Automatic Today" (CR-0004) had to define a pinned/unpinned distinction
  precisely because Calendar-as-navigation implies a live default state
  that a manual navigation action should be able to override.
