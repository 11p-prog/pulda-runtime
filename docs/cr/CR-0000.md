# CR-0000 — Pulda AI Collaboration Protocol

Version: 0.3 (extended with the Phase 0 control plane, new-chat bootstrap,

## Purpose

Define the operating model for Pulda OS during the MVP phase.

------------------------------------------------------------------------

# Roles

## User (Product Owner)

Responsible for: real-world use, decision making, product direction,
validation, feedback, final approval, and **verification** — a CR is not
Closed until the user has verified it (see Status Lifecycle below).

## ChatGPT (Chief Architect)

Responsible for: product architecture, UX philosophy, domain model,
documentation, Change Requests (CR), Architecture Decision Records (ADR),
Requests for Comments (RFC), review of Replit implementation, long-term
roadmap. Does NOT claim implementation unless verified.

## Replit (Runtime Engineer)

Responsible for: implementation, refactoring, testing, deployment, runtime
stability, Notion dashboard updates, documentation updates after
implementation (Git is the only place the CR content itself lives —
Replit edits `docs/cr/CR-XXXX.md` in place, cumulatively, never forking a
new file for the same CR).

------------------------------------------------------------------------

# Document System

Pulda's product process runs on three document types, each with a
distinct purpose. All three live under `docs/` in Git (canonical, GitHub
is the source of truth for their content).

## 1. CR (Change Request) — "무엇을 바꿀 것인가"

A directive to change something. One file per CR at `docs/cr/CR-XXXX.md`,
edited cumulatively (never forked into `CR-XXXX-v2.md`). Format:

- CR Number
- Title
- Background
- Problem
- Goals
- Requirements
- Acceptance Criteria
- Future Considerations
- Affected Components
- Status
- Related CR
- Related Notion
- Related Git Commit

## 2. ADR (Architecture Decision Record) — "왜 이렇게 설계하기로 결정했는가"

Records a design decision and its reasoning, so future contributors don't
re-litigate settled tradeoffs. Lives at `docs/adr/ADR-XXXX.md`. Format:
Title, Status, Context, Decision, Consequences.

## 3. RFC (Request for Comments) — "아직 결정되지 않았지만 함께 검토할 주제"

An open topic under discussion, not yet a build directive — e.g. AI memory
structure, personal-data sensitivity layers, the import engine design.
Lives at `docs/rfc/RFC-XXXX.md`. An RFC graduates into one or more CRs (and
often an ADR recording the decision it converged on) once a direction is
chosen; it is never implemented directly as an RFC.

# Source of Truth

## Notion — Operating DNA and approved decisions

Notion preserves Theory, Constitution, roles, architecture, approved meaning,
decisions, human corrections, and the operating dashboard.

## Git — Implementation lineage

Git preserves versioned CR acceptance criteria, ADR/RFC records, prompts,
schemas, migrations, source code, tests, commits, and releases.

## Runtime

The running application and its private data store preserve Events, AI
interpretations, human corrections, actions, outcomes, and execution evidence.

## Version History / Development Source

No store replaces the others. See `docs/governance/OPERATING-MODEL.md` for
authority, conflict resolution, and synchronization rules.

------------------------------------------------------------------------

# Status Lifecycle

Unified across CR/ADR/RFC:

```
Draft → Review → Approved → In Progress → Implemented → User Verified → Closed
```

Side states as needed: `Blocked`, `Rejected`, `Deferred`.

**Critical rule: "Implemented" is not "Closed."** Replit marking a CR
Implemented means the code is done and tested by Replit. It must still
pass through **User Verified** before it can become **Closed**. Replit
must never set a CR to Closed on its own — only to Implemented, and then
wait for the user's verification to move it further.

------------------------------------------------------------------------

# Communication Flow

Human intent or Runtime evidence → Notion DNA/decision → approved CR → Git
implementation and tests → Runtime execution → human correction/outcome →
reusable correction → next comparable interpretation → reconciliation.

Exporting a review or Markdown file is not synchronization. The previous
correction must affect future behavior and all status surfaces must agree.

## Mandatory bootstrap and continuity

Every new Pulda chat or Builder task begins by reading the relevant Notion DNA,
active CR, Git implementation/tests, Runtime evidence, and any newer emergency
patch or explicit user decision. It records confirmed, conflict, missing, and
assumption entries before planning. Material conflicts stop implementation.

This rule is enforced by `PROJECT_INSTRUCTIONS.md`, root `AGENTS.md`, and the
`pulda-governance` Skill.

# Product Principles

1. Event First — everything begins as an Event.
2. Context First — users work inside Context Workspaces.
3. Calendar is Navigation — calendar selects dates; it is not the primary
   workspace.
4. Capture First — capture immediately, interpret later.
5. Review Continuously — the system continuously learns from user history.
6. Privacy First — sensitive information must remain under user control;
   AI providers are replaceable; vendor lock-in must be avoided.

------------------------------------------------------------------------

# Historical Data

Google Docs, journals and legacy records are not attachments. They become
historical Events after import. The timeline should become searchable and
reusable. (Design direction only — see RFC-0003 / `docs/IMPORT_PIPELINE.md`;
not yet a CR in progress.)

------------------------------------------------------------------------

# Acceptance

The MVP succeeds when the user naturally records events, immediately finds
previous records, understands today's context, reviews yesterday's work,
and trusts Pulda with long-term life management.

Do not optimize for feature count. Optimize for trust.
