# CR-0003 — Recent Events

## Background
Without a way to immediately see what was just captured, users can't
verify their own input landed correctly — a core trust requirement for an
Event-first system.

## Problem
Prioritized/filtered views (like the plan card) hide low-priority or
just-captured items. There was no mandatory, chronological "what actually
happened / was captured" view.

## Goals
Give the user a always-visible, chronological log of recent Events,
scoped to their current workspace.

## Requirements
- Mandatory panel (not optional/collapsible away), always shown in the
  center workspace.
- Chronological order (by capture time), not importance/urgency-sorted —
  this is a log, not a priority queue.
- Click-to-expand for full detail (context, timestamps, importance/
  urgency) without leaving the page.
- Scoped to the active (domain, date) workspace per CR-0002 — a historical
  date shows that day's record, not a rolling "last 15 regardless of date"
  list.

## Acceptance Criteria
- Recent Events updates immediately after a capture, in the same
  workspace context the capture happened in.
- Viewing a historical date shows that date's events, not today's.

## Future Considerations
- Search/filter within Recent Events is a plausible future extension but
  is not required for this CR.

## Affected Components
`pulda/service.py::context_workspace` (`recent`), `pulda/templates/index.html`
(Recent Events section).
