# CR-0005 — Historical Import Framework

## Background
The user has a year of Google Docs journal entries that represent real
personal history (family, health, work, decisions) — not attachments, but
a "memory seed" that should become real, structured Events in Pulda.

## Problem
Importing this is a large, multi-source, privacy-sensitive feature: it
needs extraction, timeline reconstruction (dating fragments correctly onto
the date axis from CR-0002), classification, and a decision on how much of
it ever touches an external AI provider. Building it as a one-off "Google
Docs importer" would make every later source (Markdown, Notion Export,
TXT, PDF, future sources) a bolt-on special case.

## Goals
Design (not yet build) a provider-independent Import Pipeline: source
adapters are the only source-specific layer; extraction, timeline
reconstruction, classification, and Event creation are shared across all
sources.

## Requirements (design-time only — no implementation yet)
- Adapter contract: `list_sources()` / `extract(source_ref) ->
  list[RawDocument]`. Nothing downstream should know which provider a
  `RawDocument` came from.
- Timeline reconstruction assigns real dates to fragments so imported
  Events land correctly on the date axis (CR-0002).
- Classification reuses the existing rule-based classifier — per explicit
  user decision, no external AI call for extraction/classification in
  this phase.
- Imported Events are tagged `source="import:<provider>"` for
  auditability, and use the same `create_event` path as live capture — no
  new storage model.
- Any future step that would send data to an external AI provider must
  pass through the sensitivity gate defined in CR-0006 / RFC-0002.

## Acceptance Criteria (for the eventual build, not this phase)
- A new source type can be added as a single new adapter file, without
  touching timeline/classification/linking stages.
- Imported history is distinguishable from live captures and re-runnable
  per source without creating duplicates.

## Future Considerations
- Google Docs source specifically needs an OAuth/connector decision (not
  yet made) — manual paste/export vs. live API integration.
- Duplicate-import detection needs an idempotency key design.
- Bulk-imported Events likely need a review/correction UI before being
  treated as first-class history, since classification on old journal
  text will be noisier than live single-line capture.

## Affected Components
None yet — design only. See `docs/IMPORT_PIPELINE.md` for the full
pipeline shape and `docs/rfc/RFC-0003.md` for the open design questions.

## Status

checklist and January–July PDFs are the first operational memory required by
CR-0013. Implementation remains blocked until the Phase 0 control plane and
privacy/export/restore prerequisites are merged and verified.
