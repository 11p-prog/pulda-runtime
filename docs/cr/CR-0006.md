# CR-0006 — Privacy-first Architecture

## Background
Pulda's data model is explicitly designed to eventually hold family,
finance, medical, work, decisions, journals, and long-term personal
history. At commercialization scale this becomes a large concentration of
personal data, so privacy must be a first-class design constraint, not a
feature added later.

## Problem
No policy currently governs which data may be sent to external AI
providers, how sensitive fields are protected at rest, or how a user could
fully export/leave the system. Without this, later AI features (including
CR-0005's import pipeline) have no guardrail to build against.

## Goals
Establish the privacy principles and concrete guardrails future features
must satisfy, and document the current state honestly (including gaps).

## Requirements
- Document 8 principles: local-first where practical, explicit AI-provider
  control, permission-based AI access, sensitivity classification before
  external transmission, provider independence, future local-LLM support,
  encryption at rest for sensitive fields, and exportability without
  vendor lock-in.
- Audit current external data flows (Notion sync, GitHub sync) against
  these principles and record gaps honestly rather than claiming
  compliance that doesn't exist yet.
- Define the concrete mechanism required before any feature sends Event
  data to an external AI provider: check `role`/`area` against a
  configurable sensitivity list before sending.

## Acceptance Criteria
- `docs/PRIVACY_ARCHITECTURE.md` exists, covers all 8 principles, and
  states current state vs. gaps (not implemented) explicitly.
- Any future CR involving an external AI call must be checked against
  this document's guardrails before being marked Implemented.

## Future Considerations
- Encryption at rest for sensitive fields is not implemented — tracked as
  a gap, to be scheduled as its own CR when finance/medical/credential
  fields become first-class columns.
- Local-LLM support is prepared for (provider-agnostic interface
  recommendation) but not built. See RFC-0001.

## Affected Components
`docs/PRIVACY_ARCHITECTURE.md` (new). No runtime code changed by this CR
itself — it is a policy/architecture document.
