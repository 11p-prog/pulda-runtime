# CR-0007 â€” Generic File Attachments (Import Framework, Phase 1)

## Background
The user records real-world events using Google Docs journals, contracts,
and photos, and wants everything eventually pulled into Pulda (CR-0005 /
RFC-0003). Rather than waiting for the full historical-import pipeline
design to be finalized, the user asked to start with the smallest useful
piece and grow it incrementally.

## Problem
CR-0005 (Historical Import Framework) covers bulk, provider-specific
import (Google Docs journal, etc.) and is intentionally Deferred pending
open RFC-0003 questions (OAuth vs manual export, dedup, review UI). None
of that blocks a much smaller, source-agnostic capability: letting the
user attach a file (PDF, photo) directly to an Event they capture today.

## Goals
Let a single file (PDF or image) be attached to an Event at capture time
and be viewable/downloadable later, without pulling in any of CR-0005's
bulk-import machinery.

## Requirements
- New `attachments` table: one row per file, linked to an `event_id`.
- Capture bar gains an optional file picker; if a file is present on
  submit, it is saved and attached to the newly created Event.
- Files are stored on disk under a random name (never the client-supplied
  filename) with a size cap (`PULDA_MAX_ATTACHMENT_MB`, default 20MB).
- Attachments are downloadable via `/attachments/{id}`, served with their
  original filename.
- Event list views (recent events, full table) show an attachment
  indicator/link per Event.

## Acceptance Criteria
- Uploading a PDF or image through the capture bar creates an Event and
  an attachment row, and the file round-trips byte-for-byte on download.
- An Event with no attachment renders unchanged (no empty attachment UI).
- Oversized uploads are rejected with a clear error, not a silent
  truncation.

## Future Considerations
- This is intentionally *not* CR-0005: no OCR/text extraction, no
  timeline reconstruction, no bulk import, no classification of file
  contents. It only gets a file next to an Event.
- Multiple attachments per Event are supported by the schema but the UI
  has only been exercised with one at a time so far.
- When CR-0005 eventually resumes, its bulk importer should reuse this
  same `attachments` table/storage convention instead of inventing a
  second one.

## Affected Components
`pulda/db.py` (`attachments` table), `pulda/config.py`
(`attachment_dir`/`max_attachment_mb`), `pulda/service.py`
(`add_attachment`/`list_attachments`/`get_attachment`), `pulda/app.py`
(`/capture` file handling, `/attachments/{id}` download), `pulda/templates/index.html`
(capture bar file picker, attachment indicators in event lists).
