# CR-0008 — External Audit Fixes + Verification Standard

## Background
Once the repository was made readable (public, plain URL fetch), an
external audit (ChatGPT Chief Architect, cloning the 41-file repo at
commit `92d0ffc`) reviewed the actual code against CR-0001~0007's stated
requirements — the first time the CRs were checked against real code by
an independent reader rather than by the same agent that wrote them.

## Problem
The audit found the underlying work standard, not just isolated bugs: CR
docs were being marked `Implemented` based on manual/screenshot checks
that did not correspond to what the code actually did, and `Related Git
Commit` was routinely left as `(recorded at merge time)` and never filled
in. Ten concrete issues were found, several of them real behavioral bugs
directly contradicting the CR they were claimed to satisfy.

## Audit findings and resolution

| # | Finding | Resolution |
|---|---------|------------|
| 1 | Historical-date view's "전체 Event" table showed the whole context's events, not just that date's, contradicting CR-0002. | `context_workspace()` now returns date-scoped `events` for a historical date; only the live Today view shows the full context. |
| 2 | When "today" had no captures yet, "최근 Event" silently fell back to the last 15 events overall (any date), contradicting CR-0003's "historical date shows that day's record, not a rolling list" — same defect on live Today. | `recent` is now always day-scoped for the selected date, no fallback; an empty day shows an empty list. |
| 3 | Event creation and file-save/attach were not transactional — a failed upload (e.g. over size limit) left a real, permanently empty Event behind. | `/capture` now rolls back (`delete_event`) the just-created Event if attachment validation or save fails. |
| 4 | No server-side MIME/extension validation on upload (UI `accept` is not a security boundary); download response had no anti-sniffing/inline-execution protection. | Added an extension + content-type allow-list (`_validate_upload`), `X-Content-Type-Options: nosniff`, and explicit `content_disposition_type="attachment"` on download. |
| 5 | CR-0001~0007 were marked `Implemented` on 5 total automated tests; several claims (date scoping, upload validation, transactional rollback) had none. | Added targeted tests for all four fixes above (10 tests total, up from 5); see "Verification standard" below for the policy fix. |
| 6 | CR-0007 body said the Notion dashboard row was "to be added" while `STATUS.md` already listed it as present. | Not independently re-verified this pass — flagged as open until Notion is checked directly (see Future Considerations). |
| 7 | `pytest` failed to find the `pulda` package unless run with `PYTHONPATH=.`. | Added `pytest.ini` (`pythonpath = .`); `pytest` now runs standalone. |
| 8 | `requirements.txt` listed every dependency twice. | Deduplicated. |
| 9 | `classify()` could produce a `due_date` of tomorrow with a `scheduled_at` of *today* at the stated hour (e.g. "내일 오후 3시") — a self-contradictory record. | `scheduled_at` now anchors to the same calendar day as `due_date`. |
| 10 | Privacy-first (CR-0006) implements policy/documentation only; no runtime sensitivity classification, egress blocking, encryption at rest, or per-user access control exist yet. | Confirmed accurate; CR-0006 already disclosed this. No change — recorded here for traceability. |

## Verification standard (the actual fix)
The root problem was not any single bug but how "Implemented" got applied.
Going forward:
- A CR may be marked `Implemented` only when its acceptance criteria are
  backed by an automated test that would fail if the behavior regressed —
  not by a screenshot or a manual click-through alone.
- `Related Git Commit` must be filled with the actual commit hash at the
  time status changes to `Implemented`, not left as a placeholder.
- When an audit (external or self) finds a CR's code does not match its
  written requirements, the CR's `Status` is corrected in the same change
  that fixes the code — it does not stay `Implemented` while a known gap
  is open.
