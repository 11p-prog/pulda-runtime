# CR-0010 — Remove Stray "health" Link (Scoped Interim Fix)

## Background
User feedback on the workspace-tab screenshot (2026-07-13): the tab bar
should eventually work like Replit's own tab strip — tabs the user adds
and removes freely — not a fixed set of categories (오늘/Church/
Family/Knowledge). That is a real architectural redesign, explicitly
**deferred**: the user asked to narrow this round of work down to two
small, concrete items and wait for further instruction before touching
the tab model itself.

## Scope for this CR (explicitly narrowed by the user)
1. "오늘" stays the default tab — already true, no change needed.
2. Every other tab should, at minimum, follow the same date rule 오늘 already
   follows: selecting a date shows that date's screen.
3. Remove the unlabeled "health" link in the top-right header — the user
   flagged it as unidentified UI that leads to a broken-looking page.

## Findings
- Church/Family (`role:성당`, `role:가족`) already respected the selected
  date — `context_workspace()`'s general path filters by
  `selected_date` for both, going back to CR-0002/CR-0003. No change
  needed there.
- Knowledge (`ctx == "knowledge"`) returned a hardcoded empty workspace
  regardless of `selected_date` — the one tab that didn't follow the date
  rule. It has no capture path yet (sidebar already labels it "준비중"),
  so there's no real per-date data to show, but the *shape* of the
  response now explicitly carries `selected_date`/`is_today` through the
  same code path as every other tab instead of being a special case that
  ignores the date argument.
- The "health" link (`/api/health`) was a developer debug leftover: it
  left the SPA entirely to show a raw JSON blob, which reads as a broken
  page to an end user. Removed from the header; the endpoint itself still
  exists for direct/ops use, it's just no longer linked from the UI.

## Explicitly out of scope (deferred, per user instruction)
- Replit-style user-managed tabs (add/remove/rename) replacing the fixed
  role/knowledge category tabs. Waiting for the user's next instruction
  before starting this.
