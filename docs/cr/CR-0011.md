# CR-0011 — User-Managed Workspace Tabs (Replit-Style Add/Remove)

## Background
Follow-up to CR-0010. The user flagged that the center workspace tab bar
(오늘/Church/Family/Knowledge) reads as a fixed, hardcoded category list
that visually and conceptually collides with the right-sidebar "Life
Balance" breakdown (업무/가족/건강/성장), which is also role-derived. The
ask: make the tab bar user-managed like Replit's own tab strip — the user
opens and closes tabs freely — with "오늘" as the one permanent default.

## What changed
- New `workspace_tabs` table (`pulda/db.py`) replaces the old hardcoded
  `CONTEXT_TABS` list in `service.py`. Each row is `(ctx, label, icon,
  removable, sort_order)`. "오늘" is seeded non-removable; every other tab
  is user-manageable.
- `service.list_workspace_tabs()` / `add_workspace_tab()` /
  `remove_workspace_tab()` — read/add/remove, guarding "오늘" against
  removal and rejecting duplicate `ctx`s.
- UI (`templates/index.html`): each tab now has a hover-revealed × to
  close it (confirms first — closing a tab never deletes its Events, it
  only hides the tab), and a `+` opens a small picker to open a new tab.
- `min-w-0` added to the tab-bar container — without it, a wide tab row
  silently overflowed past the main column and rendered underneath the
  fixed right sidebar (a real bug hit while testing this feature, not
  hypothetical: a classic flex-item-doesn't-shrink-below-content issue).

## What's offered when adding a tab (draft, not final)
The "+" picker suggests presets inferred from the app's existing role
vocabulary (`classifier.ROLE_KEYWORDS`: 가족/회사/성당/개인) plus the
Knowledge placeholder — not a new invented taxonomy:
- Church (`role:성당`), Family (`role:가족`) — already existed pre-CR-0011.
- Work (`role:회사`), Personal (`role:개인`) — added as a draft suggestion;
  the classifier already sorts events into these roles, but no tab
  previously surfaced them.
- Knowledge (`knowledge`) — still the pre-existing placeholder, no capture
  path yet.
- Free-form: typing a project name opens a `project:<name>` tab (same
  mechanism that already auto-opens a tab once an Event uses that
  project).

This is explicitly a first draft seed, per the user's request to infer
something reasonable from the existing docs/role model — expect it to be
revisited once real usage shows which tabs people actually want open by
default.

## Explicitly out of scope this round
- Renaming a tab once created, or reordering via drag.
- Resolving the deeper Life Balance vs. tab-bar conceptual overlap the
  user raised — this CR makes the tab *bar* user-managed, it does not
  redesign the right-sidebar Life Balance widget. Flagged for the user to
  decide whether Life Balance should be reworked separately.
