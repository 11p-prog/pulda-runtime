# CR-0017 — Event Date, State, Source, and Recurrence

## Decision source

User Runtime evidence and corrections recorded in Notion on 2026-07-20.

## Implemented scope

- Calendar-selected dates are stored as `occurred_on`; `captured_at` preserves
  the real input time.
- `inbox` and `planned` migrate to `recorded`; `legacy_status` preserves the
  previous value for audit and rollback.
- `source` remains independent of lifecycle status (`manual_ui`, API,
  ChatGPT daily capture, connector/import namespaces).
- Recurrence series are stored separately from materialized occurrence
  exceptions. Date windows are calculated on demand instead of inserting an
  unbounded future.
- Event-list selection and bulk status change replace status choice during
  capture. Every changed Event still produces status history and audit rows.

## Migration and rollback

The additive migration creates `captured_at`, `occurred_on`, and
`legacy_status`. Existing timestamps backfill the first two fields. Only
`inbox`/`planned` are mapped to `recorded`, with the exact former value copied
to `legacy_status`. Rollback can restore those rows from `legacy_status` after
using the deployment platform's private database backup/export. No database,
backup, or real Event data belongs in public Git.

## Acceptance evidence

Automated tests cover selected-date capture, separate capture time, default
status/source, bulk updates, recurrence windows and occurrence exceptions.
Deployment backup/restore and Runtime user verification remain required.
