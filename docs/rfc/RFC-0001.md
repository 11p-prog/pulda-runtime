# RFC-0001 — AI Memory Structure

## Status
Open for discussion — not yet a build directive.

## Summary
As Pulda accumulates Events (live capture + eventual historical import),
what structure should its "memory" take so that future AI features
(insights, decision support, life-balance analysis) can reason over it
consistently, across providers, without re-deriving context every time?

## Questions to resolve
- Is memory just "the `events` table" queried fresh each time, or does
  Pulda need a derived/summarized memory layer (e.g. rolling per-role
  summaries) to avoid re-processing full history on every AI call?
- How does memory interact with the sensitivity classification in RFC-0002
  — does a "memory layer" ever store AI-summarized versions of sensitive
  content, and if so, under what encryption/access rules?
- Should memory be provider-specific (tied to whichever AI backend
  produced it) or provider-agnostic (a shared representation any provider
  can read/write), per CR-0006's provider-independence principle?
- How does imported historical data (CR-0005) get folded into memory
  differently from live-captured Events, if at all?

## Non-goals (for now)
This RFC does not propose a specific vector store, embedding model, or
local-LLM runtime. It is scoped to *what memory should represent*, not
*what stores it*.

## Next steps
Needs a decision conversation before it becomes a CR. Likely candidate for
an ADR once a direction (e.g. "derived summaries per role, refreshed
daily, provider-agnostic JSON") is chosen.
