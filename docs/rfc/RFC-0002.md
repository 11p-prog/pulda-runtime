# RFC-0002 — Personal Data Sensitivity Layers

## Status
Open for discussion — not yet a build directive.

## Summary
CR-0006 established the principle that Event data must be classified
before any external transmission, and that sensitive fields should
eventually be encrypted at rest. This RFC is about the concrete design of
that classification/encryption layer, which hasn't been decided yet.

## Questions to resolve
- What is the sensitivity taxonomy? A first cut might be per `role`/`area`
  (재무/건강/가족 flagged sensitive by default), but is that granular
  enough, or does it need per-field sensitivity (e.g. a 재무 Event's amount
  vs. its free-text description)?
- Is the sensitivity flag a fixed default the user can override per
  event/session, or fully user-configurable from the start?
- Encryption at rest: field-level (e.g. `cryptography` Fernet on specific
  columns) vs. whole-database encryption vs. deferring until
  finance/medical/credential become first-class columns instead of
  free-text fields?
- Where does the key live — Replit Secrets only, or does the user need an
  export/rotation path independent of the running app (relevant to the
  exportability/no-lock-in principle)?
- How does this interact with CR-0005's import pipeline — bulk-imported
  journal content is exactly the kind of data this taxonomy needs to
  protect from day one of import, not retrofitted after.

## Non-goals (for now)
Not proposing a specific encryption library version or a UI for managing
sensitivity settings — those follow once the taxonomy itself is agreed.

## Next steps
Needs a decision conversation before it becomes a CR (likely CR-0007 or
later, gating CR-0005's actual implementation). The resulting decision
should be recorded as an ADR.
