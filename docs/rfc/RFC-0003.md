# RFC-0003 — Import Engine Design

## Status
Open for discussion — not yet a build directive. Companion to CR-0005
(which is Deferred) and `docs/IMPORT_PIPELINE.md` (which holds the
provider-adapter shape already sketched).

## Summary
`docs/IMPORT_PIPELINE.md` already sketches a provider-independent pipeline
(source adapter → raw text extraction → timeline reconstruction →
classification → linking → searchable history). This RFC is where the
still-open questions from that document should be discussed before any of
it becomes a CR that Replit implements.

## Questions to resolve
- Google Docs specifically: OAuth/connector-based live access, or does the
  user export/paste content manually for the first version? (No decision
  made yet — this blocks CR-0005 resuming.)
- Duplicate-import detection: what idempotency key (source_type,
  source_ref, fragment offset — or something else) prevents re-running an
  import from creating duplicate Events?
- Review/correction UI: bulk-imported Events will have noisier
  classification than live single-line capture (per the user's own rule-
  based-classifier-only decision in CR-0005) — does the user need a
  triage/correction pass before imported Events count as first-class
  history?
- How does this interact with RFC-0002 (sensitivity layers) — journal
  import is the highest-volume, highest-sensitivity data this pipeline
  will ever process.

## Non-goals (for now)
Not proposing specific parser libraries (PDF/Markdown/etc.) — that is
implementation detail once the pipeline shape and the questions above are
settled.

## Next steps
Resolve the open questions here (and in RFC-0002, since they're coupled),
then convert into one or more concrete CRs. CR-0005 stays Deferred until
then.
